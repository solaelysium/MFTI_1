## Постановка задачи

### Бизнес-контекст

Банк формирует персональные кредитные офера для клиентов: предлагает лимиты, процентные ставки и условия погашения. Однако даже хорошо рассчитанный офер не приносит результата, если клиент его отклоняет. Задача состоит в том, чтобы заранее оценить вероятность согласия, чтобы оптимизировать условия и снизить затраты на нерелевантные коммуникации.

### ML-постановка

Требуется оценить условную вероятность согласия клиента:

$$\hat{p}_i = P(y_i = 1 \mid X_i)$$

где $y_i \in \{0, 1\}$ — факт принятия офера, $X_i$ — вектор признаков заявки, клиента, продукта и макроэкономического контекста.

Основная метрика соревнования — **ROC-AUC**:

$$\text{ROC-AUC} = P\!\left(\hat{p}_{i^+} > \hat{p}_{i^-}\right)$$

где $i^+$ и $i^-$ — случайно выбранные положительная и отрицательная заявки. Метрика инвариантна к выбору порога и корректно работает при сильном дисбалансе классов.

## Данные

- **Train:** 145 241 заявка, период 2024-02-01 -- 2025-06-05
- **Test:** 36 311 заявок, период 2025-06-05 -- 2025-12-01
- **25 сырых признаков:** условия офера (`offered_rate`, `loan_amount_last`, лимиты), кредитная история (`sum_deb_ul_*`, `cnt_deb_*`), корпоративная активность (`corp_*`, `count_all_corp_dashboard_events`), дата решения
- **Дисбаланс классов:** доля согласий около 7-9%
- **Временной разрыв:** test полностью лежит после train, что исключает случайный split и требует строгой временной валидации

## Ключевые выводы из анализа данных

### Пропуски несут сигнал

20 из 25 признаков содержат пропуски. Механизм пропусков -- MNAR (Missing Not At Random): отсутствие данных само по себе характеризует клиента. Клиент без истории овердрафтов скорее всего никогда не пользовался этим продуктом.

Для каждого столбца $j$ с пропусками было вычислено:

$$\Delta_j = \bar{y}^{\,\text{miss}}_j - \bar{y}^{\,\text{present}}_j$$

Разница доли согласий между строками с пропуском и без него:

| Признак | Пропусков (train) | $\Delta_j$ |
|---|---|---|
| `overdraft_app_term_max_360` | 96% | -0.258 |
| `loan_rev_max_start_non_fin` | 91% | -0.157 |
| `loan_rev_min_start_fin` | 86% | -0.120 |
| `sum_deb_ul_*` | 37-42% | -0.065 до -0.069 |

Полнота профиля клиента сильнее связана с согласием, чем точные числовые значения. По этой причине модели подаются бинарные индикаторы `{col}_isna` и агрегат `missing_count`, а не заполненные числа.

### Продуктовый сегмент

Признак `db_group_last` (тип последнего кредитного продукта) является одним из сильнейших предикторов. Сегмент `overdraft` показывает acceptance rate около 37% при глобальном среднем 7-9% -- разница почти в 5 раз. Клиенты с активным овердрафтом уже вовлечены в кредитные отношения и значительно чаще принимают новые офера. При этом 38% строк в train не имеют этого признака -- для них заведена отдельная категория `__MISSING__`.

### Календарные паттерны

Анализ в `exp.ipynb` показал неравномерность acceptance rate по дням недели и неделям месяца. Воскресные заявки (`is_sunday`) демонстрируют отклонение от среднего: по всей видимости, в воскресенье в офисы обращаются клиенты с более осознанным намерением. Добавление флага `is_sunday` дало прирост ROC-AUC на сырых данных: с 0.7628 до 0.7647 (+0.002). Первая и последняя недели месяца также различаются по поведению клиентов, однако соответствующие признаки не прошли отбор из-за нестабильности на valid.

### Временной дрейф

Между train и test существует разрыв около 6 месяцев. Признаки, основанные на абсолютном уровне процентных ставок (`offered_rate`, `cb_rate`, `rate_ratio`, `rate_spread`), заметно дрейфуют. Для измерения дрейфа применялся Population Stability Index:

$$\text{PSI} = \sum_{k=1}^{K} \left(p^{\text{test}}_k - p^{\text{train}}_k\right) \cdot \ln\frac{p^{\text{test}}_k}{p^{\text{train}}_k}$$

`rate_ratio` дополнительно нестабилен при `cb_rate` близком к нулю в части test-записей. Все ratio-признаки исключены из финального набора.

### Кластеризация поведенческих профилей

KMeans на поведенческих признаках (кредитная активность, цифровое взаимодействие) хорошо разделяет архетипы клиентов в пространстве PCA. Тем не менее таргет размазан внутри кластеров: кластер описывает профиль клиента в целом, а не его готовность принять конкретный офер прямо сейчас. Кластерные признаки не улучшили качество модели и в финал не вошли.

## Эксперименты

В `exp.ipynb` были проверены следующие гипотезы:

| Гипотеза | Результат |
|---|---|
| Ratio-признаки (`rate_ratio`, `loan_to_limit_*`) | Ухудшают valid и public AUC из-за дрейфа |
| Флаг `is_sunday` | +0.002 ROC на сырых данных |
| Top-5 `_isna` флагов | +0.001 ROC дополнительно к `is_sunday` |
| `missing_share` + все sparse `_isna` | Незначительный прирост, больше шума |
| Импутация числовых пропусков (XGB/Ridge) | Нейтрально или хуже: сигнал «пусто/не пусто» размывается |
| Сегментация разреженных признаков на 2-3 бина | Не даёт прироста поверх XGB с нативным NaN |
| Кластерные признаки и MoE по сегментам | Хуже baseline |
| Optuna-тюнинг гиперпараметров | Маргинальный прирост, не превзошёл blend |
| OOF temporal ensemble | Переобучение на train-периоде, public score хуже |

Ключевой вывод из `exp.ipynb`: интерпретация через SHAP и permutation importance подтверждает, что модель в первую очередь опирается на **полноту профиля клиента** (missingness-блок), а затем на параметры офера и кредитную историю. Ложноположительные ошибки -- это преимущественно «полные профили, но отказ»: клиенты с хорошей историей, не принявшие конкретные условия.

## Финальное решение

### Feature engineering

Базовый конвейер `ratio_reduced` строит из 25 сырых признаков расширенный набор:

1. Календарные признаки из `decision_day`: день недели, месяц, флаги выходных.
2. Бинарные индикаторы пропусков `{col}_isna` для каждого числового столбца.
3. Счётчики `missing_count` и `missing_share`.
4. Частотное кодирование категорий: `db_group_last_freq`, `fl_adminarea_freq`.
5. Вычисляемые ratio (rate_spread, loan_to_limit и др.) -- добавляются для анализа, затем часть дропается.

Финальный набор **`reduced_v2_no_ratios`** получается удалением нестабильных и слабых признаков. Итого: **82 признака**.

### Валидация

Применяется жёсткое временное разделение:

$$\mathcal{D}_{\text{train}} = \{i : d_i < \texttt{2025-02-01}\}, \quad \mathcal{D}_{\text{valid}} = \{i : d_i \geq \texttt{2025-02-01}\}$$

Это соответствует реальному сценарию: модель обучается на прошлом и применяется к будущим заявкам.

### Greedy blend

Три модели (XGBoost, CatBoost, LightGBM) объединяются жадным ансамблем. На каждом шаге выбирается кандидат и вес $w^*$, максимизирующие ROC-AUC на valid:

$$\hat{p}^{(t)} = w^* \cdot \hat{p}^{(t-1)} + (1 - w^*) \cdot \hat{p}_{\text{candidate}}$$

$$w^* = \arg\max_{w \in [0.1,\, 0.9]} \;\text{ROC-AUC}\!\left(y_{\text{valid}},\; w \cdot \hat{p}^{(t-1)} + (1 - w) \cdot \hat{p}_{\text{candidate}}\right)$$

Модели вносят некоррелированные ошибки: XGBoost и LightGBM работают с числовыми признаками по-разному, CatBoost обрабатывает категории нативно. Их комбинация снижает дисперсию итогового предсказания.

### Результаты

| Метрика | Valid | Public LB |
|---|---|---|
| ROC-AUC | **0.768359** | ~0.767314 |
| PR-AUC | **0.329786** | - |
